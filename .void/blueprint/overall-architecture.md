# 总体架构

## 文档定位
本文档是 `blueprint` 层的唯一总体架构说明，用来定义系统目标、边界、核心对象、能力分层以及主数据/控制流。

它只回答“系统整体如何组织与衔接”，不展开各子域实现细节；具体技术方案落在 `solution` 层对应文档中。

## 目标与核心价值
本项目的目标不是一次性生成某份视频分析结果，而是形成一个可持续运行的网球复盘系统。

理想闭环应简化为：
1. 你录制训练或比赛视频。
2. 系统自动完成识别、接入、切片、精选、分析与沉淀。
3. 你在 `Obsidian` 中查看带有历史上下文的复盘结果。

系统真正追求的不是单点分析能力，而是低干预闭环能力：把一次训练从原始素材转化为可追踪、可复用、可持续积累的知识资产。

## 系统边界
### 用户负责
- 录制训练或比赛视频。
- 保证素材具备基本可分析条件，例如画面可见、时长足够、主要回合未被持续遮挡。

### 系统负责
- 识别新的网球训练或比赛会话。
- 判断视频与会话的归属关系，并把素材接入本地处理环境。
- 将长视频转换为可分析的片段集合。
- 从片段中筛选少量最有代表性的样本。
- 结合长期上下文生成本次复盘结论。
- 将结果沉淀到 `Obsidian`，并更新后续可复用的训练记忆。

### 外部依赖与运行边界
- `Apple Watch / iPhone`：提供运动信号和视频素材入口。
- `iCloud Photos / Photos Library`：承载视频资产管理与同步。
- `MacBook`：承担编排、处理、模型调用和本地落盘。
- 外部模型服务：承担多模态理解、归因分析和建议生成。
- `Obsidian`：作为最终知识沉淀与后续检索入口。

重计算与编排主要发生在 `MacBook` 侧；移动端主要作为信号源和素材入口。

## 核心对象
### Session
表示一次训练或比赛，是全链路的顶层聚合对象。

它把运动识别、视频归属、片段分析、报告沉淀与记忆更新串成同一个事实单元。

### Source Video Asset
表示照片库中的原始视频资产。

它最初只是候选素材，经过归属判断和同步完成后，才成为某个 `Session` 的确定输入。

### Rally Clip
表示从原始视频中切出的回合级或近似回合级片段集合。

它是系统从连续视频流转向结构化分析的基础中间对象。

### Representative Clip
表示从 `Rally Clip` 中进一步筛选出的重点分析样本。

它承担“用有限分析预算覆盖最大信息量”的作用。

### Analysis Package
表示一次分析任务的输入封装，通常包括代表片段、检索出的历史上下文以及分析约束。

### Observation
表示系统对某个片段、模式或整次会话形成的结构化判断。

它既可以来自模型直接输出，也可以来自后处理后的归纳整理。

### Memory
表示跨 `Session` 保留的稳定训练认知，例如反复出现的问题、阶段目标、已验证有效的改进方向和长期变化趋势。

它不是原始聊天记录，而是服务于持续复盘的可更新知识层。

### Session Note
表示写入 `Obsidian` 的最终知识载体。

它承载本次会话的结论、证据引用、行动建议，以及与历史状态的衔接关系。

## 核心关系
- 一个 `Session` 可以关联一个或多个 `Source Video Asset`。
- 一个 `Source Video Asset` 会被转换为多个 `Rally Clip`。
- `Rally Clip` 中只有少量样本会进入 `Representative Clip` 集合。
- `Representative Clip` 与 `Memory` 共同组成 `Analysis Package`，驱动 `Observation` 生成。
- `Observation` 被整理后沉淀为 `Session Note`。
- `Session Note` 和本次结论会更新 `Memory`，影响后续 `Session` 的分析上下文。

系统因此形成“会话 -> 素材 -> 片段 -> 结论 -> 记忆 -> 下一次会话”的递归闭环。

## 总体分层
### 第一层：会话发现层
负责回答“是否发生了一次值得处理的网球活动”。

主输出：
- 候选训练窗口
- 网球倾向
- 会话置信度

对应子方案：
- [运动识别架构](../solution/01-sport-detection.md)

### 第二层：素材归属与接入层
负责回答两个问题：
- 哪些视频属于这次 `Session`
- 这些视频如何稳定到达本地处理环境

主输出：
- 归属于 `Session` 的视频资产
- 已同步并导出到本地工作目录的原始视频

对应子方案：
- [视频识别架构](../solution/02-video-identification.md)
- [视频同步架构](../solution/03-video-sync.md)

### 第三层：片段结构化层
负责把原始长视频转换为可枚举、可筛选、可引用的片段集合。

这一层的重点不是一次性完美理解视频，而是先把“难以直接处理的连续流”变成“可操作的结构化素材”。

主输出：
- 回合级或近似回合级片段
- 片段级基础元数据

对应子方案：
- [视频切片架构](../solution/04-video-segmentation.md)

### 第四层：代表性选择层
负责从大量片段中选出最值得重点分析的少量样本。

这一层承担两个作用：
- 用有限分析预算覆盖最高信息密度素材
- 避免人工随机选段造成的偏差

对应子方案：
- [视频精选架构](../solution/05-video-selection.md)

### 第五层：分析与知识沉淀层
负责把代表片段转成可复盘、可继承的知识产物。

这一层包含三个连续环节：
- 分析：形成结构化观察、问题判断、亮点识别和训练建议
- 记忆：把本次结论压缩成可复用的长期上下文
- 沉淀：将结果写入 `Obsidian`，形成稳定事实源

对应子方案：
- [视频分析架构](../solution/06-video-analysis.md)
- [结果保存架构](../solution/07-result-persistence.md)

## 主数据流
1. `Apple Watch / iPhone` 输出候选训练窗口和相关信号。
2. 照片库基于时间窗口召回候选视频，并结合内容复核确定归属。
3. 确认归属的视频通过照片库同步到 `MacBook`，再导出到本地工作目录。
4. 系统对长视频执行切片，得到 `Rally Clip` 集合。
5. 系统从候选片段中精选少量 `Representative Clip`。
6. 分析编排层检索与当前阶段相关的 `Memory`，并组装 `Analysis Package`。
7. 模型服务输出 `Observation`，系统进一步整理为结构化结论。
8. 结果写入 `Obsidian`，形成 `Session Note`。
9. 本次结论被压缩回写到 `Memory`，供后续会话再次检索。

## 主控制流
这条链路不是一次性脚本，而是由多个阶段性决策组成的控制流。

关键控制点包括：
- 是否形成了足够可信的 `Session`
- 是否找到可归属的视频资产
- 视频是否稳定到达本地
- 切片结果是否足够可用
- 精选结果是否覆盖足够多的训练维度
- 分析输出是否满足结构化契约
- 是否成功形成单一且可追踪的 `Session Note`

任一关键控制点失败，都应进入降级路径，而不是让整次 `Session` 直接丢失。

## 真值与降级策略
系统中的“真值”不是单一来源，而是分层形成的：

- 训练发生的真值：优先来自显式 `Tennis workout`，其次来自被动识别与轻确认。
- 视频归属的真值：来自时间召回与内容复核的组合。
- 片段价值的真值：来自质量过滤、代表性、多样性与训练价值约束。
- 分析结果的真值：应被视为“结构化观察 + 推断 + 证据”，而非绝对事实。
- 最终事实源：写入 `Obsidian` 的 `Session Note`。

系统设计应允许“部分自动 + 局部确认”的现实路径，不要求每一层都百分之百自动且百分之百正确。

## 架构原则
### 自动化优先
除视频采集外，其余环节默认由系统自动触发和推进。

### 先结构化，再智能化
先稳定对象和链路，再逐步提升识别、筛选和分析质量。

### 记忆不是聊天记录
系统维护的是面向训练复盘的长期认知层，而不是依赖单次对话历史保存上下文。

### 结果必须可追踪
每次输出都应能回溯到对应会话、对应片段与对应历史背景，避免只有结论、没有依据。

### 渐进增强
首要目标是端到端闭环成立，切片精度、精选质量和分析深度可以逐步演进。

## 架构演进方向
### 第一阶段：闭环成立
先跑通“采集 -> 接入 -> 粗切片 -> 基础分析 -> 报告入库”。

### 第二阶段：代表性提升
减少随机性，让被选中的片段更能代表问题、亮点和近期训练主题。

### 第三阶段：记忆稳定
形成跨周、跨月的训练画像，使报告体现连续性和阶段判断。

### 第四阶段：教练化体验
让系统不仅描述问题，也能围绕当前训练阶段给出更稳定、更个性化的建议。

## 总体判断
这套系统的关键不只是模型能力，而是链路编排能力：
- 前段要尽量自动发现与归属
- 中段要把长视频压缩成高信息密度片段
- 后段要把分析结果稳定沉淀为可复用知识

因此，系统成败更依赖阶段边界是否清晰、对象契约是否稳定、降级路径是否明确，而不只是某个单点算法是否更强。
