# 总体架构（Blueprint）

## 目标与边界
本系统的目标是把一次网球训练或比赛，转化为一条尽量自动的复盘链路：
- 自动发现训练发生
- 自动定位对应视频
- 自动把视频送到 `Mac`
- 自动切出适合复盘的片段
- 自动精选少量代表性片段
- 自动生成结构化分析
- 自动保存到 `Obsidian`

本蓝图只描述系统边界、能力分层与主要数据流，不展开各子域的技术细节。具体实现方案分别落在对应的子架构文档中。

## 系统边界
系统跨越四个运行边界：
- `Apple Watch / iPhone`：负责训练发现与视频来源入口
- `iCloud Photos / Photos Library`：负责视频同步与资产管理
- `MacBook`：负责视频消费、处理、编排与模型调用
- `Obsidian`：负责最终知识沉淀与后续复用

这四个边界中，重计算与编排主要发生在 `MacBook` 侧；`Apple Watch / iPhone` 更适合作为信号源和素材入口。

## 核心对象
系统围绕 5 个核心对象运作：

### 1. Session
表示一次训练或比赛，是整个链路的主对象。

它从“运动识别”开始形成，在“结果保存”阶段成为稳定的事实单元。

### 2. Source Video Asset
表示照片库中的原始视频资产。

它与 `session` 的关系一开始只是候选归属，经过识别与同步后，才变成确定输入。

### 3. Clip
表示从原始视频中切出的 point 级或紧凑聚合片段。

它是后续精选和分析的直接输入。

### 4. Analysis Package
表示一次模型分析请求的输入集合，通常包括：
- 精选后的片段
- 检索到的上下文
- prompt 任务约束

### 5. Session Note
表示落在 `Obsidian` 中的最终事实源。

它承载这次训练的结构化结论、证据引用和后续行动项。

## 能力分层
总体上可以分成四层。

### 第一层：训练发现层
负责回答“是否发生了一次值得处理的网球 session”。

主输出：
- 候选训练窗口
- 网球倾向
- 置信度

对应子架构：
- [运动识别架构](../solution/01-sport-detection.md)

### 第二层：素材归属与到达层
负责回答两个问题：
- 哪些视频属于这次 session
- 这些视频如何稳定到达 `Mac`

主输出：
- 归属于 session 的视频资产
- 已导出到本地工作目录的原始视频

对应子架构：
- [视频识别架构](../solution/02-video-identification.md)
- [视频同步架构](../solution/03-video-sync.md)

### 第三层：视频处理层
负责把长视频转成“可被模型消费”的少量高价值片段。

这一层分两个连续子域：
- 先切片，回答“哪些时间段是一分或紧凑分点”
- 再精选，回答“哪些片段最值得分析”

对应子架构：
- [视频切片架构](../solution/04-video-segmentation.md)
- [视频精选架构](../solution/05-video-selection.md)

### 第四层：分析与沉淀层
负责把精选片段转成可复盘的知识产物。

这一层分两个连续子域：
- 分析：生成结构化结论
- 保存：把结论沉淀为稳定可引用的笔记

对应子架构：
- [视频分析架构](../solution/06-video-analysis.md)
- [结果保存架构](../solution/07-result-persistence.md)

## 主数据流
系统的主数据流如下：

1. `Apple Watch / iPhone` 输出候选训练窗口。
2. 照片库基于训练窗口召回候选视频，并做内容复核。
3. 确认归属的视频通过照片库同步到 `Mac`，再导出到本地工作目录。
4. `Mac` 对长视频执行切片，得到 point 级或紧凑聚合片段。
5. 从候选片段中精选少量代表性样本。
6. 分析编排层组装 `Analysis Package` 并调用模型。
7. 结果写入 `Obsidian`，形成 `Session Note`。
8. 后续分析再次从历史 `Session Note` 中检索上下文，形成闭环。

## 主控制流
这条链路不是一次性脚本，而是由多个阶段性决策串起来的控制流。

关键控制点包括：
- 是否已形成足够可信的 `session`
- 是否找到了可归属的视频资产
- 视频是否已稳定到达 `Mac`
- 切片结果是否足够可用
- 精选结果是否覆盖足够多的训练维度
- 模型输出是否满足结构化契约
- 是否成功落盘为单一 `Session Note`

任一关键控制点失败，都应优先进入降级路径，而不是让整个 session 丢失。

## 回退与真值策略
系统中的“真值”不是单一来源，而是分层形成的：

- 训练发生的真值：优先来自显式 `Tennis workout`，其次来自被动识别与轻确认
- 视频归属的真值：来自时间召回与内容复核的组合
- 片段价值的真值：来自质量过滤、代表性、多样性和训练价值约束
- 分析结果的真值：不是绝对事实，而是“结构化观察 + 推断 + 证据”
- 最终事实源：写入 `Obsidian` 的 `Session Note`

系统设计上应允许“部分自动 + 局部确认”，不要求每一层都百分之百自动且百分之百正确。

## 部署职责
不同设备承担不同职责：

### `Apple Watch / iPhone`
- 采集运动信号
- 形成训练窗口
- 提供照片库资产入口

### `MacBook`
- 监听同步结果
- 导出原始视频
- 执行切片、精选、分析编排
- 写入 `Obsidian`

### 外部模型服务
- 消费 `Analysis Package`
- 输出结构化分析结果

### `Obsidian`
- 承载最终知识沉淀
- 为后续 session 提供可检索上下文

## 架构判断
这套系统的关键不是单点模型能力，而是链路编排：
- 前段要尽量自动发现与归属
- 中段要把长视频压缩成高信息密度片段
- 后段要把分析结果稳定沉淀成可复用知识

因此，系统成败更依赖“阶段边界是否清晰、对象契约是否稳定、降级路径是否明确”，而不只是某一个模型或算法是否更强。
