# 视频切片方案

## 文档定位
本文档定义“长视频切成回合级或紧凑回合级小视频”的当前方案选择、实现边界和演进路径。

它回答的是：
- 在当前这台 `Mac` 上，视频切片应该如何落地。
- 哪些技术方案现实可行。
- 当前版本选择哪条主路径，为什么。

它不回答的是：
- 最终哪些片段值得进入分析。
- 如何基于片段生成复盘结论。
- 云端训练、重模型实验或跨平台部署细节。

## 问题定义
输入是比赛或训练时录制的长视频。单个文件通常较长，且包含大量非回合内容，例如：
- 发球前准备。
- 回合间停顿。
- 捡球。
- 走位和换边。
- 休息。

目标是把长视频切成多个规整的小视频文件，输出两类产物：
- `point clip`：只包含一个完整回合，语义范围是“从开球到死球”。
- `compact clip`：把多个相邻且间隔很短的 `point clip` 聚合成更紧凑的复盘片段。

当前系统的第一性目标不是逐帧级边界精度，而是显著减少手工切片工作量，同时保证回合语义尽量完整。

## 当前执行环境约束
当前方案必须服从本地执行环境，而不是假设理想训练环境。

已确认条件：
- 运行平台是 `macOS` 的 `Apple Silicon`。
- 本地已安装 `ffmpeg` 和 `ffprobe`。
- 本地 `python3` 可用，但当前环境没有预装 `numpy`、`opencv`、`librosa`、`mediapipe`、`torch` 等依赖。
- 不应把首版方案建立在 `CUDA`、`Linux` 或大规模训练环境上。

由此得到的设计约束：
- 首版主路径必须能在“仅有 `ffmpeg` + Python 标准库”的条件下运行。
- 首版不依赖重型视觉模型、姿态模型、球轨迹模型或时序深度学习框架。
- 如果后续需要增强，应优先考虑少量本地依赖，而不是直接引入研究型模型栈。

## 方案比较
### 方案 A：`FFmpeg-first` 零重依赖方案
用 `ffmpeg` 和 `ffprobe` 提取音频与时间结构信号，再用 Python 规则状态机决定边界，最后由 `ffmpeg` 导出小视频。

优点：
- 当前机器可直接运行。
- 依赖最少，部署成本最低。
- 可解释、可调试、可快速形成闭环。

缺点：
- 视觉理解能力弱。
- 边界精度主要受音频和时间规则限制。
- 对噪声环境较敏感。

适用性判断：
- 这是当前推荐主方案。

### 方案 B：轻量多模态增强方案
在方案 A 基础上增加少量本地依赖，例如 `numpy`、`opencv-python`、`librosa`，把音频规则扩展为“音频 + 低成本视觉运动量”的融合切片。

优点：
- 比纯音频更稳。
- 仍然可以在当前 `Mac` 上本地执行。
- 不需要进入重模型或训练流程。

缺点：
- 需要补安装依赖和维护虚拟环境。
- 调参复杂度高于方案 A。

适用性判断：
- 这是当前明确的升级路线。

### 方案 C：语义增强或学习型方案
引入姿态、球轨迹、时间动作分割、`GEBD` 或个性化监督学习模型。

优点：
- 理论精度上限更高。
- 在积累足够标注后，可以持续贴合你的录制习惯。

缺点：
- 对环境、依赖、数据和调试能力要求明显更高。
- 与当前“本机先跑通”的目标不匹配。

适用性判断：
- 当前不进入主路径，只保留为未来实验方向。

## 当前决策
当前采用方案 A 作为主方案。

决策原因：
- 它符合当前 `Mac` 的真实执行条件。
- 它能最先解决“手工切片太费时”的核心问题。
- 它能保留清晰的升级路径，不会把架构锁死在重模型或特定平台上。

因此，当前视频切片层的主线不是“多模态全开”，而是：
1. 用 `ffprobe` 和 `ffmpeg` 从长视频中提取低成本信号。
2. 用规则状态机生成保守但完整的回合边界。
3. 用重编码方式导出精确切点的小视频。
4. 在输出层同时保留 `point clip` 和 `compact clip`。

## 总体架构
### Stage 1：素材探测
目标是先拿到可用于切片决策的基础事实。

输入：
- 单个长视频文件。

输出：
- 视频时长。
- 帧率、分辨率、音频流存在性。
- 供后续处理使用的统一时间基准。

实现建议：
- 使用 `ffprobe` 获取容器和流信息。
- 如果视频没有音频流，则直接进入视觉缺省降级路径。

### Stage 2：低成本信号提取
目标是从长视频中提取足够便宜、足够稳定的候选切片信号。

当前主路径只使用以下信号：
- 音频静默区间。
- 音频能量变化。
- 候选活跃段长度。
- 邻近候选段之间的间隔长度。

实现建议：
- 用 `ffmpeg` 的 `silencedetect` 提取长静默窗。
- 可选地用 `astats`、`volumedetect` 或音频包络统计辅助判断音频活跃度。
- 所有信号按时间轴对齐到秒级或亚秒级窗口。

这一阶段不试图“懂网球动作”，只负责构造候选边界证据。

### Stage 3：候选活跃段构建
目标是从低成本信号里拼出“疑似在打球”的时间段。

基本思路：
- 长静默窗通常对应回合间停顿、捡球、休息或换边。
- 非静默且持续时间落在合理范围内的区间，优先视为候选活跃段。
- 过短区间先视为噪声，不直接形成 `point clip`。

这里采用保守策略：
- 宁可把一个点切得略长，也不要把一个点切碎。
- 对候选活跃段前后增加固定缓冲时间。

### Stage 4：规则状态机切分
目标是把候选活跃段转换成符合网球语义的 `point clip`。

当前状态机建议使用以下状态：
- `pre_serve`
- `rally`
- `dead_ball`
- `break`

状态机的核心职责：
- 区分短暂停顿和真正的回合结束。
- 把发球前准备保留在当前回合起点附近。
- 把死球后的短尾巴保留在当前回合终点附近。
- 用滞回逻辑避免一分被切成多个碎片。

关键规则建议：
- 只有当静默或低活跃状态持续超过阈值，才允许结束当前点。
- 对持续时间明显过短的候选段，优先并入前后点而不是单独输出。
- 对持续时间异常长的候选段，先标记为低置信度，而不是强行再切。

### Stage 5：导出 `point clip`
目标是把状态机确定的时间范围导出为单独视频文件。

当前环境下的关键决策：
- 不采用仅基于关键帧的粗切。
- 采用重编码导出，优先保证切点语义正确。

实现建议：
- 使用 `ffmpeg -ss ... -to ...` 逐段导出。
- 视频编码优先使用 `h264_videotoolbox`，利用本机硬件加速。
- 音频可使用 `aac`。
- 输出文件名中带序号和时间范围，便于回溯。

这样做的原因是：
- 关键帧切割会让起止点偏移。
- 当前任务更关心边界正确性，而不是无损无重编码。

### Stage 6：生成 `compact clip`
目标是把若干相邻 `point clip` 聚合成更适合复盘的片段。

聚合条件建议：
- 相邻点之间的空档很短。
- 点与点之间语义连续。
- 聚合后总时长仍适合观看和后续分析。

聚合原则：
- `point clip` 是基础事实，不应被覆盖。
- `compact clip` 是派生产物，可以随阈值重复生成。

## 输出契约
### `point clip`
每个 `point clip` 至少保留以下字段：
- `clip_id`
- `source_video_id`
- `start_time`
- `end_time`
- `duration`
- `point_index`
- `confidence`
- `boundary_evidence`
- `export_path`

### `compact clip`
每个 `compact clip` 至少保留以下字段：
- `compact_clip_id`
- `source_video_id`
- `start_time`
- `end_time`
- `duration`
- `child_point_clip_ids`
- `aggregation_reason`
- `export_path`

### `boundary_evidence`
建议记录：
- 使用了哪些信号。
- 主要静默窗位置。
- 主要阈值。
- 是否发生了降级。

这里的核心要求是可追踪，而不是字段越多越好。

## 阈值策略
首版不应追求自动学习阈值，而应采用可理解、可人工调整的显式参数。

建议参数包括：
- `min_silence_duration`
- `min_point_duration`
- `max_point_duration`
- `pre_roll_seconds`
- `post_roll_seconds`
- `merge_gap_seconds`
- `break_gap_seconds`

策略要求：
- 参数应集中放入配置，而不是散落在代码里。
- 参数默认值应偏保守，先减少误切，再追求更紧。

## 降级路径
### 无音频流
如果视频没有音频流，则当前主方案无法可靠切点。

此时应降级为：
- 仅按固定时间窗口粗切。
- 或直接标记为“需要增强版切片器处理”。

### 音频质量差
如果环境噪声大、邻场声音多、风噪明显，则切点置信度下降。

此时应降级为：
- 放宽边界。
- 增大前后缓冲。
- 减少过短点的独立输出。

### 候选段异常
如果出现明显过长或明显过短的候选段：
- 不强行精确切分。
- 输出低置信度标记，交由后续抽查。

降级策略的原则是：
- 宁可保守切片，也不要生成看似精确但语义错误的边界。

## 当前明确不做
当前版本不纳入以下能力：
- 球轨迹检测。
- 姿态估计。
- 计分牌识别。
- `GEBD`。
- 深度学习时序动作分割。
- 依赖 `PyTorch` 或 `CUDA/MPS` 的训练型边界检测器。

这些方向不是永久排除，只是不进入当前这台 `Mac` 的主执行路径。

## 升级路线
### V1：本地闭环版
依赖：
- `ffmpeg`
- `ffprobe`
- Python 标准库

能力：
- 音频静默检测
- 规则状态机切点
- `point clip` 导出
- `compact clip` 聚合

目标：
- 先把手工切片工作量显著压下来。

### V2：轻量增强版
新增依赖：
- `numpy`
- `opencv-python`
- `librosa`

新增能力：
- 音频 onset 和节律特征
- 低帧率视觉运动量
- 音频与视觉联合决策

目标：
- 降低纯音频方案在噪声场景下的误切率。

### V3：个性化增强版
前提：
- 已积累你自己手工修正过的切片结果。

新增能力：
- 个性化阈值学习
- 轻监督时间窗分类
- 更稳的边界精修

目标：
- 让切片器逐步适应你的机位、球场和录制习惯。

## 验证计划
当前阶段的验证重点不是学术指标，而是是否真的减少手工劳动。

至少应观察以下指标：
- 单个长视频中，成功导出的 `point clip` 数量是否合理。
- 是否经常把一个回合切成两个。
- 是否经常把两个回合粘在一起。
- 发球前准备和死球后收尾是否经常被切掉。
- 每小时素材所需人工修正时间是否显著下降。

建议采用一组固定样本视频做回归验证，并对每次参数调整记录前后差异。

## 实施判断
在当前这台 `Mac` 上，视频切片层的正确策略不是一开始追求最强模型，而是先构建一个可运行、可解释、可迭代的本地切片器。

因此，当前的架构判断是：
- 主方案选 `FFmpeg-first`。
- 精确导出优先于无重编码。
- `point clip` 优先于 `compact clip`。
- 轻量增强优先于重模型实验。

后续如果真实样本证明纯音频路线不够稳，再进入 `opencv + librosa` 的增强路径，而不是直接跳到研究型方案。
